# Container-Orchestrierung und Deployment-Strategien

## 1. Warum braucht man Container-Orchestrierung?

In echten Produktionsumgebungen von Unternehmen laufen oft **Hunderte oder Tausende von Containern** gleichzeitig. Ohne Automatisierung stößt man sofort auf folgende Probleme:
* **Ausfall:** Was passiert, wenn ein Server abstürzt, auf dem 50 Container laufen?
* **Skalierung:** Wie bewältigt man einen plötzlichen Ansturm von Usern (z. B. am Black Friday)?
* **Routing:** Wie finden die Container untereinander den Weg und wie wird der Datenverkehr der Kunden gleichmäßig verteilt?
* **Updates:** Wie aktualisiert man eine App auf Version 2.0 ohne Ausfallzeiten (Downtime)?

Die **Container-Orchestrierung** löst diese Probleme. Sie fungiert wie der Dirigent eines Orchesters: Sie sorgt dafür, dass alle einzelnen Instanzen (Container) perfekt zusammenspielen, zur richtigen Zeit starten und die Anwendung durchgehend verfügbar bleibt.

---

## 2. Wie funktioniert Container-Orchestrierung?

Die Orchestrierung basiert auf dem Prinzip des **deklarativen Modells** (Soll-Zustand vs. Ist-Zustand):

1. **Die Deklaration (Der Wunsch):** Der Entwickler schreibt eine Konfigurationsdatei (meist im YAML-Format). Darin steht: *"Ich möchte, dass von der App 'Web-Shop' immer genau 3 Instanzen laufen."*
2. **Der Abgleich (Der Control Loop):** Das Orchestrierungstool liest diesen Wunsch (**Soll-Zustand**) und prüft kontinuierlich die Realität (**Ist-Zustand**).
3. **Die automatische Korrektur:** Wenn ein Server ausfällt und nur noch 2 Instanzen laufen, merkt das System das sofort. Es startet vollautomatisch eine neue Instanz auf einem anderen, gesunden Server, um den Soll-Zustand wiederherzustellen (*Self-Healing*).

Zudem kümmert sich das Tool um das **Scheduling** (die kluge Verteilung auf die Server) und das **Load Balancing** (die gleichmäßige Verteilung des Datenverkehrs).

---

## 3. Bekannte Orchestrierungs-Technologien

* **Kubernetes (K8s):** Der absolute Marktführer und Industriestandard. Ursprünglich von Google entwickelt, heute Open Source. Extrem mächtig und flexibel, aber auch sehr komplex.
* **Docker Swarm:** Das native Orchestrierungstool von Docker. Wesentlich einfacher einzurichten als Kubernetes, bietet dafür aber weniger Funktionen für riesige Architekturen.
* **Managed Services (Cloud-Varianten):** Da das Aufsetzen von Kubernetes schwer ist, nutzen die meisten Firmen die verwalteten Dienste der Cloud-Anbieter: **Amazon EKS**, **Google GKE** oder **Azure AKS**.

## 4. Horizontale Skalierung (Scaling Out)

Im Cloud-Computing unterscheidet man zwei Arten der Skalierung:

* **Vertikale Skalierung (Scaling Up):** Man gibt dem *bestehenden* Server mehr Power (mehr CPU, mehr RAM). *Nachteil:* Irgendwann ist das physische Limit erreicht und der Server muss oft kurz neugestartet werden.
* **Horizontale Skalierung (Scaling Out):** Man fügt *weitere* Server oder Container desselben Typs hinzu. 

Im Kontext von Cloud- und Container-Anwendungen ist die **horizontale Skalierung** der Standard. Steigt der Traffic auf einer Website, startet Kubernetes innerhalb von Sekunden vollautomatisch 10 weitere Container der App (*Auto-Scaling*). Ein vorgelagerter **Load Balancer** verteilt die Nutzeranfragen gleichmäßig auf alle Container. Sinkt der Traffic nachts, werden die überflüssigen Container gelöscht, um Kosten zu sparen.

## 5. Deployment-Strategien

Wenn eine neue Version einer Anwendung (v2) eingespielt werden soll, gibt es verschiedene Strategien, um das Risiko von Ausfällen zu minimieren:

### A. Rolling Update (Standard)
Die Container der Version 1 werden schrittweise durch Version 2 ersetzt. Erst wenn ein neuer Container (v2) erfolgreich gestartet ist, wird ein alter Container (v1) abgeschaltet.
* **Vorteil:** Keine Downtime für den Nutzer.
* **Nachteil:** Für eine gewisse Zeit laufen Version 1 und Version 2 gleichzeitig.

### B. Blue-Green Deployment
Es existieren zwei identische Umgebungen. "Blue" ist die aktuelle Live-Umgebung (v1). In der isolierten Umgebung "Green" wird die neue Version (v2) hochgefahren und getestet. Ist alles bereit, wird der Load Balancer mit einem Klick komplett von Blue auf Green umgeschaltet.
* **Vorteil:** Nahezu null Downtime und ein extrem schnelles Rollback (Zurückschalten auf Blue), falls Fehler auftreten.
* **Nachteil:** Doppelte Infrastrukturkosten während des Prozesses.

### C. Canary Deployment
Die neue Version (v2) wird nur für einen ganz kleinen Teil der Nutzer (z. B. 5 %) freigeschaltet. Der Rest verbleibt auf v1. Läuft alles stabil, wird der Anteil schrittweise auf 100 % erhöht.
* **Vorteil:** Schlägt das Update fehl, sind nur sehr wenige Nutzer betroffen.
* **Nachteil:** Hoher Aufwand bei der Konfiguration des Netzwerk-Routings.

### D. Recreate
Alle alten Container (v1) werden gleichzeitig abgeschaltet. Erst danach werden die neuen Container (v2) hochgefahren.
* **Vorteil:** Einfachste Methode. Es laufen niemals zwei Versionen gleichzeitig (wichtig bei komplexen Datenbank-Updates).
* **Nachteil:** Es entsteht eine spürbare Downtime für die Nutzer, während die neuen Container booten.
