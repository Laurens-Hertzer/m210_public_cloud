# Container-Technologie und Virtualisierung im Vergleich

## 1. Was ist Container-Technologie?

Die Container-Technologie ist eine Methode, um Anwendungen inklusive all ihrer Abhängigkeiten (Bibliotheken, Konfigurationsdateien) in einer isolierten Einheit – dem **Container** – zu verpacken. 

Im Gegensatz zur Hardware-Virtualisierung wird kein kompletter Computer simuliert. Alle Container teilen sich den **Betriebssystem-Kernel des Host-Systems** und werden über Mechanismen wie Linux-*Namespaces* und *cgroups* voneinander isoliert.

---

## 2. Vor- und Nachteile gegenüber VMs

### Vorteile von Containern:
* **Geringer Ressourcenverbrauch:** Kein eigenes Betriebssystem nötig, minimaler RAM- und CPU-Overhead im Leerlauf.
* **Blitzschneller Start:** Startet in Millisekunden statt Minuten.
* **Hohe Effizienz:** Höhere Packungsdichte (mehr Anwendungen auf derselben Hardware).
* **Perfekte Portabilität:** Läuft auf dem Entwickler-Laptop exakt wie in der Cloud.

### Nachteile von Containern:
* **Geringere Isolation (Sicherheit):** Da der Kernel geteilt wird, kann ein Kernel-Exploit potenziell alle Container auf dem Host gefährden.
* **Betriebssystem-Abhängigkeit:** Ein Linux-Container benötigt zwingend einen Linux-Kernel (ohne zusätzliche Emulationsschichten).

## 3. Bekannte Produkte

| Bereich | Produkte für Virtuelle Maschinen (VMs) | Produkte für Container |
| :--- | :--- | :--- |
| **Software / Runtimes** | VMware vSphere/ESXi, Hyper-V, VirtualBox, KVM | **Docker**, Podman, containerd |
| **Orchestrierung** | VMware vCenter, OpenStack | **Kubernetes (K8s)**, Docker Swarm |
| **Cloud-Dienste** | AWS EC2, Azure VMs, Google Compute Engine | AWS EKS, Google GKE, Azure AKS |

---

## 4. Technologischer Vergleich

| Kriterium | Virtuelle Maschine (VM) | Container |
| :--- | :--- | :--- |
| **Betriebssystem** | Eigenes, vollständiges Gast-Betriebssystem (Guest OS). | Teilt sich den Kernel des Host-Betriebssystems. |
| **Bereitstellung** | Dauert Minuten (Booten des OS). | Dauert Millisekunden bis Sekunden (Prozessstart). |
| **Speicherplatz** | Groß (Gigabyte-Bereich pro VM). | Sehr klein (Megabyte-Bereich). |
| **Effizienz** | Geringer (Overhead durch Hardware-Emulation). | Extrem hoch (direkte Ausführung auf dem Kernel). |

---

## 5. Können VMs immer durch Container ersetzt werden?

**Nein.** In folgenden Szenarien sind VMs weiterhin notwendig:
1. **Unterschiedliche Kernel:** Ausführen einer Windows-Legacy-App auf einem Linux-Server.
2. **Maximale Sicherheitsisolation:** Strikte Hardware-Trennung (z. B. bei Multi-Tenant-Systemen im Bankenbereich).
3. **Direkter Hardware-Zugriff:** Apps, die exklusive Kontrolle über spezifische Treiber benötigen.

*Praxis-Tipp:* In der Cloud werden Container meist *innerhalb* von VMs gestartet, um beide Vorteile zu kombinieren.

---

## 6. Kernbegriffe: Image, Container, Registry

* **Image (Das Rezept):** Eine schreibgeschützte (Read-Only) Vorlage, die den Code und die Laufzeitumgebung enthält.
* **Container (Der Kuchen):** Die lebendige, laufende Instanz eines Images mit einer zusätzlichen, beschreibbaren Schicht für den Betrieb.
* **Registry (Das Kochbuch):** Der zentrale Lagerplatz (Repository) für Images, von dem aus sie hochgeladen (*push*) oder heruntergeladen (*pull*) werden (z. B. Docker Hub).

---

## 7. Das Prinzip der Unveränderlichkeit (Immutable Infrastructure)

In der Praxis werden laufende Container **niemals im Betrieb verändert oder gepatched**. 
Wenn ein Update oder eine Konfigurationsänderung ansteht:
1. Ein neues **Image** wird gebaut.
2. Der alte Container wird **gelöscht**.
3. Ein neuer Container wird auf Basis des neuen Images gestartet.

Dies sorgt für **Reproduzierbarkeit** (kein "Configuration Drift") und einfache **Skalierbarkeit**. Wichtige Daten werden über externe *Volumes* außerhalb des Containers gespeichert.

---

## 8. Unterschied: Self-Managed vs. Fully Managed

### Self-Managed (Eigenverantwortlich)
* **Merkmal:** Sie mieten nackte VMs und installieren/warten das Container-Cluster (z. B. Kubernetes) komplett selbst.
* **Vorteile:** Volle Kontrolle über jede Stellschraube, kein Vendor Lock-in.
* **Nachteile:** Hoher administrativer Aufwand (Updates, Backups, Routing, Ausfallsicherheit).

### Fully Managed (Vollständig verwaltet)
* **Merkmal:** Der Cloud-Anbieter stellt den Dienst schlüsselfertig bereit (z. B. Google GKE, AWS EKS) und wartet die Infrastruktur im Hintergrund.
* **Vorteile:** Enorme Zeitersparnis, automatische Skalierung und hohe Verfügbarkeitsgarantien.
* **Nachteile:** Höhere Kosten pro Stunde und stärkere Bindung an den jeweiligen Anbieter.
