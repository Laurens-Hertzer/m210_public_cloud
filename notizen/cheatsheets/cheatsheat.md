# Def Cloud 

### Cloud Computing

Cloud computing is a model that allows easy, on-demand access to a shared pool of IT resources (like servers, storage, or apps) over the internet. These resources can be set up or removed very quickly, with almost no effort or contact with the provider.

The model includes:

* **5 essential characteristics**
* **3 service models** (IaaS, PaaS, SaaS)
* **4 deployment models** (Private, Community, Public, Hybrid)

# Merkmale Characteristiken Cloud

1. On-demand self-service. Du kannst IT-Ressourcen wie Server, Speicherplatz oder Datenbanken selbst jederzeit bereitstellen, ohne jemanden beim Anbieter kontaktieren zu müssen. 

2. Broad network access:
zugriff von beliebigen geräten über das netzwerk/internet
Die Cloud-Dienste sind über das Internet erreichbar und funktionieren auf vielen verschiedenen Geräten.

3. Resource Pooling (Gemeinsame Ressourcennutzung)

Der Cloud-Anbieter stellt seine Hardware vielen Kunden gleichzeitig zur Verfügung. Die Ressourcen werden je nach Bedarf verteilt.

Beispiel:
Mehrere Unternehmen nutzen dieselben Rechenzentren, aber jedes Unternehmen sieht nur seine eigenen Daten und Anwendungen.

Wichtig:
Du weißt meist nicht genau, auf welchem Server deine Daten gespeichert sind. Oft kannst du nur die Region wählen (z. B. Schweiz oder Deutschland).

4. Rapid Elasticity (Schnelle Skalierbarkeit)

Die Cloud kann Ressourcen sehr schnell vergrößern oder verkleinern, je nachdem wie viel Leistung gerade benötigt wird.

Beispiel:
Ein Online-Shop hat an Black Friday plötzlich 10-mal mehr Besucher. Die Cloud stellt automatisch mehr Serverleistung bereit. Nach dem Aktionstag wird sie wieder reduziert.

5. Measured Service (Messbarer Service)

Die Nutzung der Cloud wird automatisch gemessen und überwacht. Du bezahlst meist nur das, was du tatsächlich nutzt.

Beispiel:
Wenn du 100 GB Speicher nutzt, bezahlst du für 100 GB. Wenn du später 200 GB benötigst, steigt die Abrechnung entsprechend.

6. Software as a Service (SaaS) – Software als Dienstleistung
Du nutzt fertige Programme des Anbieters, die komplett in der Cloud laufen. Du musst nichts installieren oder updaten – der Zugriff erfolgt einfach über den Webbrowser oder eine App auf deinem Gerät. Du hast keinen Zugriff auf die Server oder die IT-Infrastruktur dahinter, sondern kannst meistens nur ein paar persönliche Einstellungen anpassen.

Beispiel:
Ein webbasierter E-Mail-Dienst (wie Gmail) oder Streaming-Dienste (wie Netflix). Du loggst dich einfach über den Browser oder die App ein und nutzt den Dienst direkt, ohne dir Gedanken über die Server im Hintergrund zu machen.

Wichtig:
Du kaufst die Software nicht, sondern "mietest" nur die Nutzung. Um Wartung, Datensicherung und Updates kümmert sich komplett der Anbieter.

7. Platform as a Service (PaaS) – Plattform als Dienstleistung
Der Anbieter stellt dir eine fertige Arbeitsumgebung (eine Plattform) zur Verfügung, auf der du deine eigenen Programme oder Apps entwickeln, testen und laufen lassen kannst. Du musst dich nicht um Server, Betriebssysteme oder Speicherplatz kümmern – das regelt der Anbieter. Du kontrollierst und verwaltest nur deine selbst geschriebenen Anwendungen.

Beispiel:
Ein Software-Entwickler schreibt eine neue Smartphone-App. Er nutzt PaaS, um seinen Code hochzuladen. Die Plattform sorgt automatisch dafür, dass die App im Internet erreichbar ist und stabil läuft, ohne dass der Entwickler einen eigenen Server einrichten muss.

Wichtig:
Dieses Modell ist ideal für Programmierer. Sie sparen viel Zeit, weil sie sich nicht mit der Server-Konfiguration herumschlagen müssen, sondern sich ganz auf das Schreiben von Code konzentrieren können.

8. Infrastructure as a Service (IaaS) – Infrastruktur als Dienstleistung
Du mietest die rein "nackte" IT-Hardware in der Cloud – also Rechenleistung (virtuelle Server), Festplattenspeicher und Netzwerkkomponenten. Was du darauf installierst, bleibt komplett dir überlassen. Du bist selbst dafür verantwortlich, ein Betriebssystem (z. B. Windows oder Linux) aufzuspielen, Sicherheitsupdates zu installieren und deine Programme einzurichten.

Beispiel:
Ein Unternehmen braucht einen neuen Datenserver. Statt ein teures Gerät zu kaufen und ins Büro zu stellen, mietet es einen virtuellen Server in der Cloud und installiert dort die benötigte Software selbst.

Wichtig:
Das ist das flexibelste Modell, weil du die volle Kontrolle über das gesamte System hast. Es bedeutet aber auch den größten Aufwand, da du dich selbst um die Verwaltung und Absicherung der Betriebssysteme kümmern musst.

# Deployement Methode

9. Private Cloud (Private Cloud)
Die Infrastruktur wird exklusiv von einer einzigen Organisation genutzt. Niemand sonst hat Zugriff. Die Server stehen entweder im eigenen Unternehmen oder werden exklusiv von einem Dienstleister betrieben.

Beispiel:
Eine Großbank betreibt ein eigenes, hochsicheres Rechenzentrum nur für die internen Abteilungen.

Wichtig:
Bietet maximale Kontrolle und Sicherheit, ist aber wegen der eigenen Hardware am teuersten.

10. Community Cloud (Gemeinschafts-Cloud)
Mehrere Organisationen mit ähnlichen Interessen (z. B. gleichen Gesetzen oder Sicherheitsvorgaben) teilen sich eine Cloud. Sie ist nur für diese feste Gruppe zugänglich.

Beispiel:
Mehrere Krankenhäuser nutzen eine gemeinsame Cloud, um Patientendaten sicher nach strengen Vorgaben auszutauschen.

Wichtig:
Man teilt sich Kosten und Aufwand mit Gleichgesinnten in einer geschlossenen Umgebung.

11. Public Cloud (Öffentliche Cloud)
Die Cloud-Dienste sind für die breite Öffentlichkeit (jeden Privatnutzer oder jede Firma) frei zugänglich. Die Infrastruktur steht in den weltweiten Rechenzentren des Anbieters.

Beispiel:
Du speicherst Dateien auf Google Drive oder ein Startup mietet Server bei Amazon Web Services (AWS).

Wichtig:
Extrem flexibel und günstig (Zahlung nach Nutzung), aber man hat weniger Kontrolle über den genauen Speicherort.

12. Hybrid Cloud (Hybride Cloud)
Eine Kombination aus mehreren Modellen (meist Private und Public Cloud). Die Systeme bleiben eigenständig, können aber Daten und Apps flexibel austauschen.

Beispiel:
Ein Onlineshop speichert Kundendaten sicher in der Private Cloud, holt sich bei hohem Ansturm (z. B. Black Friday) aber Serverleistung aus der Public Cloud dazu.

Wichtig:
Das Beste aus beiden Welten: Hohe Sicherheit für sensible Daten kombiniert mit schneller Skalierbarkeit für Spitzenzeiten.

Was ist Container-Technologie?Die Container-Virtualisierung ist eine Methode, um Anwendungen inklusive all ihrer Abhängigkeiten (Code, Laufzeitumgebung, Systemwerkzeuge, Bibliotheken) in einer isolierten Umgebung – dem Container – zu verpacken.Der entscheidende Unterschied zur klassischen Virtualisierung: Container teilen sich den Kernel (das Herzstück) des Host-Betriebssystems. Sie emulieren keine komplette Hardware, sondern isolieren lediglich die Prozesse voneinander. Dadurch sind sie extrem leichtgewichtig.2. 1. Warum braucht man Container-Orchestrierung?
Ein oder zwei Container auf dem eigenen Laptop zu starten, ist kinderleicht. In echten Unternehmen laufen aber oft Hunderte oder Tausende Container gleichzeitig auf vielen verschiedenen Servern.

Ohne ein automatisches System (den "Orchester-Dirigenten") würde pures Chaos ausbrechen, weil man manuell folgendes regeln müsste:

Welcher Container läuft auf welchem Server, der noch genug Platz (RAM/CPU) hat?

Was passiert, wenn ein Server nachts um 3 Uhr abstürzt? (Container müssen woanders neu starten).

Wie finden sich die Container untereinander im Netzwerk?

Wie verteilt man den Datenverkehr der Kunden gleichmäßig auf alle Container?

2. Wie funktioniert Container-Orchestrierung?
Das wichtigste Prinzip dahinter ist das deklarative Modell (Soll-Zustand vs. Ist-Zustand):

Du definierst den Soll-Zustand: Du schreibst in eine Datei (meist YAML): "Ich möchte, dass von meiner Web-App immer genau 3 Container laufen."

Der Orchestrator überwacht den Ist-Zustand: Er prüft permanent, was gerade wirklich läuft.

Automatisches Handeln (Auto-Healing): Stürzt einer der 3 Container ab, merkt das System das sofort. Es löscht den kaputten Container und startet blitzschnell einen neuen, um den Soll-Zustand (3 Container) wiederherzustellen.

Darüber hinaus kümmert sich das System um das Scheduling (die kluge Verteilung auf die Server) und das Load Balancing (Verteilung des Datenverkehrs).

3. Bekannte Technologien
Kubernetes (K8s): Der unangefochtene Industriestandard. Extrem mächtig, von Google entwickelt, kann absolut alles, ist aber auch recht komplex zu lernen.

Red Hat OpenShift: Eine plattformübergreifende Enterprise-Version von Kubernetes mit zusätzlichen Sicherheits- und Komfort-Werkzeugen für große Firmen.

Docker Swarm: Die "kleine Schwester" von Kubernetes. Direkt in Docker integriert, viel einfacher zu bedienen, aber für riesige, komplexe Systeme oft nicht mächtig genug.

Amazon ECS (Elastic Container Service): Die AWS-eigene Alternative zu Kubernetes, wenn man sich voll und ganz im Amazon-Ökosystem bewegt.

4. Horizontale Skalierung (Horizontal Scaling)
In der Cloud unterscheidet man zwei Arten, wie man auf steigenden Besucheransturm reagiert:

Vertikale Skalierung (Scale Up): Du machst den einen vorhandenen Server größer (mehr CPU, mehr RAM). Metapher: Du kaufst einen größeren Lkw. Nachteil: Irgendwann gibt es keine größere Hardware mehr und der Server muss beim Upgrade oft kurz neu starten.

Horizontale Skalierung (Scale Out): Du fügst mehrere Instanzen desselben Typs hinzu (du klonst deine Container oder Server). Metapher: Du kaufst zehn kleine Lieferwagen statt eines Riesen-Lkw. Vorteil: Nahezu unendlich skalierbar, extrem ausfallsicher (wenn einer ausfällt, laufen die anderen weiter) und perfekt für die Cloud geeignet.

5. Deployment-Strategien
Wenn Entwickler eine neue Version ihrer Software (z. B. Version 2.0) aufspielen wollen, gibt es verschiedene Strategien, wie man die alten Container gegen die neuen austauscht:

Recreate (Neu erstellen)
Alle alten Container (V1) werden komplett abgeschaltet und erst danach werden die neuen (V2) gestartet.

Vorteil: Keine Vermischung von alten und neuen Datenstrukturen.

Nachteil: Es gibt eine spürbare Ausfallzeit (Downtime) für die Nutzer.

Rolling Update (Schrittweises Update)
Der Orchestrator ersetzt einen alten Container nach dem anderen. Während Container 1 auf V2 aktualisiert wird, nehmen Container 2 und 3 (V1) noch den Datenverkehr an.

Vorteil: Keine Ausfallzeit für den Nutzer.

Nachteil: Für eine kurze Zeit laufen V1 und V2 gleichzeitig im System – die App muss damit umgehen können.

Blue-Green Deployment (Blau-Grün)
Du hast zwei komplett identische Umgebungen. "Blau" ist live (V1). Du installierst V2 in aller Ruhe auf "Grün". Wenn alles getestet ist und funktioniert, wird der Router mit einem Klick umgeschaltet, sodass alle Nutzer sofort auf "Grün" landen.

Vorteil: Keine Downtime und ein perfektes "Rollback" (wenn V2 Fehler hat, schaltet man sofort wieder zurück auf Blau).

Nachteil: Doppelte Infrastrukturkosten während des Updates.

Canary Deployment (Kanarienvogel-Methode)
Die neue Version V2 wird erst einmal nur für einen ganz kleinen Teil der Nutzer (z. B. 5 %) freigeschaltet. Die restlichen 95 % nutzen weiter V1. Wenn nach ein paar Stunden keine Fehler im System auftauchen, wird die neue Version schrittweise für alle ausgerollt.

Vorteil: Minimiert das Risiko von Fehlern dramatisch. Falls ein Bug existiert, sind nur sehr wenige Nutzer betroffen.
